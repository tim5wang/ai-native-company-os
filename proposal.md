# ANC 产品计划：AI 原生企业业务操作系统

> **版本**: v2.0  
> **日期**: 2026-05-08  
> **定位**: 面向企业数字化负责人和数字化实施服务商的 AI 原生企业业务操作系统  
> **MVP 场景**: 资产盘点 / 仓储出入库

---

## 1. 一句话定位

ANC 是一个 **AI 原生企业业务操作系统**。

它让企业数字化负责人或实施服务商用自然语言描述业务需求，由 Builder Agent 将需求转化为可运行的业务应用、数据模型、页面、流程、权限和审计规则；运行期尽量使用确定性代码和规则执行，Agent 主要负责构建、维护、诊断和持续优化。

简化表达：

> **业务人员描述需求，AI 生成业务系统；系统确定性运行，AI 持续维护。**

---

## 2. 为什么需要 ANC

传统企业数字化不缺工具，缺的是一个能快速把业务流程落成稳定系统的底座。

### 2.1 传统 OA / ERP 的问题

1. **定制慢**  
   业务流程一变，就需要供应商或 IT 团队改表、改页面、改流程、改权限。

2. **成本高**  
   很多中小企业的软件采购成本并不高，真正贵的是实施、二开、维护和反复沟通。

3. **响应慢**  
   业务负责人知道需求，但不会建系统；IT 懂系统，但不一定懂业务细节。

4. **体验差**  
   很多系统只解决“能不能用”，没有解决“业务人员愿不愿意用”。

5. **AI 是外挂**  
   企业里的 AI 往往停留在聊天框、知识库问答或独立 Copilot，无法真正理解业务流程、数据结构、权限边界和运行状态。

### 2.2 当前 AI 工具的问题

ChatGPT、Coze、Dify、n8n、Make 等工具都很有价值，但它们没有完整解决企业业务系统生成问题：

| 类型 | 价值 | 不足 |
|---|---|---|
| Chat Assistant | 易用、灵活、适合问答和生成 | 不直接进入业务系统，缺少流程、权限、审计和运行态 |
| AI 应用平台 | 适合做问答、客服、简单 Agent | 多数运行时依赖 LLM，稳定性和成本不可控 |
| Workflow Builder | 适合确定性自动化 | 需要人先设计流程，不适合从业务需求直接生成系统 |
| 低代码平台 | 能搭表单和流程 | 仍然依赖实施人员配置，AI 原生程度低 |
| 传统 OA / ERP | 企业级能力较完整 | 僵化、定制贵、AI 难以内生集成 |

ANC 的切入点不是再做一个聊天框，也不是再做一个流程画布，而是：

> **让 AI 进入企业业务系统的构建、运行和维护链路。**

## 3. 目标用户

第一版不直接面向所有企业员工，而是面向两类用户。

### 3.1 企业数字化负责人

包括企业内部的信息化负责人、运营负责人、管理者或懂业务的数字化推进者。

他们的需求：

- 快速把线下流程变成线上应用；
- 降低对外部软件厂商和内部开发资源的依赖；
- 能看清业务流程、数据流、权限和运行状态；
- 业务变化后可以快速调整系统。

### 3.2 数字化实施服务商

包括为传统企业做 OA、ERP、进销存、仓储、制造、零售数字化的服务商。

他们的需求：

- 缩短交付周期；
- 降低定制开发成本；
- 将行业经验沉淀为可复用模板；
- 同时服务更多客户；
- 通过 AI 辅助需求澄清、方案生成、实施和运维。

## 4. 核心产品理念

### 4.1 构建期 AI，运行期确定性

ANC 不追求让 LLM 在每一次业务流程中临场判断。

更适合企业级场景的方式是：

```text
构建期：
业务需求 → Builder Agent 澄清 → 生成模型/页面/流程/权限/代码 → 测试 → 发布

运行期：
用户操作 / 业务事件 → 确定性代码和规则执行 → 日志 / 审计 / 通知 / 状态更新
```

这样可以降低：

- LLM 幻觉风险；
- 运行成本不确定性；
- 审计困难；
- 延迟和稳定性问题。

Agent 的主要职责是：

- 帮人构建系统；
- 帮人理解系统；
- 帮人修改系统；
- 帮人定位问题；
- 帮人持续优化系统。

### 4.2 产品体验像 Workspace Agent，运行机制像 Workflow Engine

用户体验上，ANC 应该像一个懂业务的数字化顾问：

- 用户用自然语言描述目标；
- Agent 主动追问缺失信息；
- Agent 给出业务对象、流程、权限和页面方案；
- 用户确认后发布应用。

运行机制上，ANC 应该像一个可靠的企业级工作流和应用平台：

- 流程可视化；
- 数据结构清晰；
- 权限可控；
- 日志可查；
- 版本可回滚；
- 异常可诊断；
- 成本可预测。

### 4.3 可复用执行，而不是 Prompt 模板

ANC 要沉淀的不是提示词，而是可运行的业务资产。

建议将这类资产定义为 **Business Skill**。

一个 Business Skill 包含：

- 数据模型；
- 表单页面；
- 列表 / 详情 / 看板页面；
- 工作流；
- 角色和权限；
- 通知规则；
- 外部连接器；
- 审计规则；
- 测试用例；
- 运维策略；
- 版本历史。

例如：

- 资产盘点 Skill；
- 仓储入库 Skill；
- 仓储出库 Skill；
- 采购审批 Skill；
- 报销 Skill；
- 员工入职 Skill；
- 合同归档 Skill。

这些 Skill 可以被 Builder Agent 生成、复用、修改和组合。

## 5. MVP 场景：资产盘点 / 仓储出入库

第一阶段选择 **资产盘点 / 仓储出入库** 作为 MVP 场景。

原因：

1. 它比普通审批流更能体现“生成业务应用”，而不只是“生成流程”。
2. 它包含企业数字化的典型要素：数据模型、移动端操作、后台管理、权限、流程、日志、审计。
3. 它适合传统企业，尤其是制造、贸易、零售、仓储、物业、学校、医院等场景。
4. 它容易展示 AI 从需求到系统的完整构建链路。

### 5.1 MVP 要解决的业务问题

企业中常见的资产和库存管理问题：

- 资产台账分散在 Excel；
- 入库、出库、领用、归还记录不完整；
- 盘点依赖人工登记；
- 实物与系统数据不一致；
- 异常缺少追踪；
- 管理者无法实时查看库存和资产状态；
- 每个企业字段、流程、权限都不同，标准软件很难完全适配。

### 5.2 MVP 应支持的核心对象

第一版建议支持以下业务对象：

1. **资产 / 物料**
   - 名称
   - 编码
   - 分类
   - 规格型号
   - 单位
   - 状态
   - 所属仓库 / 位置
   - 负责人
   - 二维码 / 条码

2. **仓库 / 存放位置**
   - 仓库名称
   - 区域
   - 货架 / 房间 / 工位
   - 管理员

3. **入库单**
   - 入库类型
   - 物料明细
   - 数量
   - 经办人
   - 审核人
   - 入库时间

4. **出库单**
   - 出库类型
   - 领用人 / 部门
   - 物料明细
   - 数量
   - 审核状态
   - 出库时间

5. **盘点任务**
   - 盘点范围
   - 盘点负责人
   - 计划时间
   - 实盘数量
   - 账面数量
   - 差异原因
   - 处理状态

6. **库存流水 / 操作日志**
   - 操作类型
   - 操作人
   - 操作对象
   - 操作前后数量
   - 时间
   - 来源单据

### 5.3 MVP 应支持的角色

- 系统管理员；
- 仓库管理员；
- 资产管理员；
- 普通员工 / 领用人；
- 审批人；
- 管理层查看者。

### 5.4 MVP 应生成的应用形态

1. **移动端 / H5 操作端**
   - 扫码查看资产；
   - 扫码入库；
   - 扫码出库；
   - 盘点录入；
   - 异常上报。

2. **Web 管理后台**
   - 资产 / 物料台账；
   - 仓库和位置管理；
   - 入库单管理；
   - 出库单管理；
   - 盘点任务管理；
   - 库存流水；
   - 异常差异处理；
   - 数据看板。

3. **Builder Agent 对话入口**
   - 需求录入；
   - 需求澄清；
   - 方案确认；
   - 应用生成；
   - 修改需求；
   - 版本发布。

### 5.5 MVP 演示闭环

MVP 最重要的不是功能数量，而是证明以下闭环可行：

```text
1. 用户描述需求
   “我们需要一个仓库出入库和资产盘点系统，支持扫码、审批和库存流水。”

2. Builder Agent 澄清需求
   - 管理哪些物品？
   - 是否需要多仓库？
   - 入库是否需要审批？
   - 出库是否需要审批？
   - 谁可以盘点？
   - 是否需要扫码？
   - 是否需要库存预警？

3. 生成结构化规格
   - 数据模型
   - 页面列表
   - 流程定义
   - 权限矩阵
   - 通知规则
   - 测试用例

4. 用户确认

5. 系统生成并发布应用
   - H5 操作端
   - Web 管理后台
   - 工作流
   - 数据表
   - 权限配置
   - 日志审计

6. 用户运行应用
   - 创建物料
   - 扫码入库
   - 发起出库
   - 审批出库
   - 发起盘点
   - 处理盘点差异

7. 用户提出修改
   “出库超过 5000 元需要部门负责人审批，否则仓库管理员直接确认。”

8. Agent 分析影响并生成变更
   - 修改流程规则
   - 更新权限
   - 生成测试
   - 创建新版本

9. 用户预览并发布
```

## 6. 第一版产品范围

### 6.1 必须做

1. **Builder Agent**
   - 自然语言需求输入；
   - 需求澄清；
   - 生成结构化业务规格；
   - 生成应用方案；
   - 支持用户修改需求。

2. **业务中间表示，Business IR**
   - 数据模型定义；
   - 页面定义；
   - 流程定义；
   - 权限定义；
   - 通知定义；
   - 审计定义；
   - 测试定义。

3. **应用生成器**
   - 根据 Business IR 生成可运行应用；
   - 支持 Web 管理后台；
   - 支持移动端 / H5 操作页；
   - 支持表单、列表、详情、基础看板。

4. **工作流引擎**
   - 支持提交、审批、驳回、完成；
   - 支持条件分支；
   - 支持角色节点；
   - 支持状态流转；
   - 支持日志记录。

5. **权限系统**
   - 用户；
   - 角色；
   - 部门；
   - 页面权限；
   - 数据权限；
   - 操作权限。

6. **日志与审计**
   - 操作日志；
   - 流程日志；
   - 数据变更记录；
   - Agent 生成记录；
   - 版本记录。

7. **版本管理**
   - 应用版本；
   - 变更说明；
   - 发布记录；
   - 基础回滚能力。

### 6.2 暂时不做

第一版不要做太多，否则会失焦。

暂时不做：

- 通用 OA 全家桶；
- 复杂 ERP；
- 财务核算；
- 复杂 BI；
- 多租户计费系统；
- 插件市场；
- 大规模连接器生态；
- 完全自主的无人值守修改生产系统；
- 复杂流程画布编辑器。

流程画布可以有只读或轻编辑能力，但不应成为 MVP 的主入口。MVP 的主入口应该是 Builder Agent。

## 7. 产品架构

### 7.1 总体结构

```text
用户 / 实施顾问
      ↓
Builder Agent
      ↓
结构化需求规格 Spec
      ↓
Business IR
      ↓
应用生成器 / 流程生成器 / 权限生成器 / 测试生成器
      ↓
Compiled Business App
      ↓
确定性运行时
      ↓
日志 / 审计 / 监控 / 反馈
      ↓
Ops Agent / Builder Agent 持续优化
```

### 7.2 核心模块

#### 7.2.1 Builder Agent

负责把自然语言业务需求变成结构化系统设计。

输入：

- 用户描述；
- 行业模板；
- 现有数据表；
- 企业组织结构；
- 历史应用配置；
- 用户反馈。

输出：

- 需求规格；
- 数据模型；
- 页面方案；
- 流程方案；
- 权限矩阵；
- 测试用例；
- 发布计划。

#### 7.2.2 Business IR

Business IR 是 ANC 的关键底层设计。

它是业务系统的中间表示，用来承接 Agent 生成结果，并驱动后续应用生成和运行。

Business IR 至少包含：

```text
App
├── Entities       数据对象
├── Fields         字段
├── Relations      关联关系
├── Pages          页面
├── Forms          表单
├── Workflows      流程
├── Roles          角色
├── Permissions    权限
├── Notifications  通知
├── AuditRules     审计规则
├── Tests          测试用例
└── Versions       版本
```

Business IR 的价值：

- 让 AI 输出可检查；
- 让应用生成可控；
- 让运行时确定性执行；
- 让版本 diff 可读；
- 让系统可审计、可回滚、可维护。

#### 7.2.3 Compiled Business App

Compiled Business App 是由 Business IR 生成的可运行业务应用。

它不是 prompt，也不是临时 Agent 任务，而是包含完整运行资产：

- 数据库表；
- API；
- 页面；
- 流程；
- 权限；
- 测试；
- 日志；
- 部署配置。

#### 7.2.4 确定性运行时

运行时负责稳定执行业务流程。

能力包括：

- 表单提交；
- 数据校验；
- 状态流转；
- 审批；
- 通知；
- 权限判断；
- 日志记录；
- 异常处理。

运行时原则：

- 优先规则和代码；
- 必要时才调用 LLM；
- 所有 LLM 参与都必须可记录、可解释、可追踪；
- 高风险操作必须人工确认。

#### 7.2.5 Ops Agent

Ops Agent 不直接替人改生产系统，而是先做诊断和建议。

能力包括：

- 分析错误日志；
- 发现流程卡点；
- 识别异常数据；
- 解释系统行为；
- 生成修复建议；
- 生成变更草案；
- 辅助测试和回归。

## 8. Agent 设计

### 8.1 Builder Agent

职责：

- 需求澄清；
- 业务建模；
- 流程设计；
- 页面设计；
- 权限设计；
- 测试生成；
- 变更影响分析。

典型对话：

```text
用户：我们想做一个资产盘点和仓库出入库系统。

Builder Agent：
我需要确认几个问题：
1. 管理的是固定资产、耗材，还是两者都有？
2. 是否有多个仓库或存放地点？
3. 入库是否需要审批？
4. 出库是否需要审批？
5. 是否需要员工扫码领用？
6. 盘点差异由谁处理？
7. 是否需要库存下限预警？
```

### 8.2 Schema Agent

职责：

- 生成数据对象；
- 生成字段；
- 生成关联关系；
- 识别枚举值和状态机；
- 生成数据校验规则。

### 8.3 Workflow Agent

职责：

- 生成流程节点；
- 生成状态流转；
- 生成审批规则；
- 生成异常分支；
- 生成通知规则。

### 8.4 UI Agent

职责：

- 生成表单；
- 生成列表；
- 生成详情页；
- 生成移动端扫码操作页；
- 生成管理后台导航。

### 8.5 Test Agent

职责：

- 根据业务规则生成测试用例；
- 覆盖正常流程；
- 覆盖异常流程；
- 覆盖权限边界；
- 覆盖数据校验。

### 8.6 Ops Agent

职责：

- 日志解释；
- 异常诊断；
- 流程瓶颈分析；
- 生成修复建议；
- 生成版本变更草案。

## 9. 关键能力优先级

- 自然语言需求 → 结构化规格；
- 结构化规格 → 数据模型；
- 数据模型 → 页面；
- 结构化规格 → 工作流；
- 角色权限生成；
- 应用发布；
- 基础日志和审计；
- 应用修改和版本记录。

### P1：增强体验

- 模板复用；
- 业务规则可视化；
- 简单流程图预览；
- 扫码能力；
- 数据导入导出；
- 企业微信 / 飞书 / 钉钉通知；
- 库存预警；
- 盘点差异报表。

### P2：形成壁垒

- Business Skill 市场；
- 行业模板库；
- 连接器生态；
- 运行数据驱动的 Agent 优化；
- Company Context；
- 多应用协同；
- 私有部署。

## 10. 路线图

### Phase 0：产品和技术设计

目标：明确最小闭环和底层抽象。

产出：

- MVP 业务流程定义；
- Business IR 初版；
- Agent 分工设计；
- 应用生成架构；
- 运行时架构；
- 权限和审计模型；
- Demo 脚本。

### Phase 1：单场景 Demo

目标：跑通资产盘点 / 仓储出入库的生成和运行。

能力：

## 12. 竞争壁垒

ANC 的壁垒不在于“能不能调用 LLM”，而在于以下几个方面。

### 12.1 Business IR

谁能把自然语言需求稳定转成可检查、可运行、可维护的业务中间表示，谁就能控制生成质量。

### 12.2 企业级运行时

企业客户需要的不只是生成页面，还需要：

- 权限；
- 审计；
- 日志；
- 版本；
- 回滚；
- 数据安全；
- 稳定运行。

### 12.3 Business Skill 积累

每交付一个场景，都可以沉淀为可复用 Skill。

随着行业模板增加，交付成本会持续下降。

### 12.4 Company Context

长期来看，ANC 需要理解企业的：

- 组织结构；
- 角色权限；
- 业务对象；
- 流程；
- 数据；
- 文档；
- 日志；
- 用户行为；
- 业务指标。

这会形成持续上下文优势。

### 12.5 实施服务商生态

数字化实施服务商可以把行业经验沉淀到 ANC 上，ANC 则帮助他们提升交付效率。这个生态一旦形成，会比单纯 SaaS 更有扩展性。

## 13. 风险与约束

### 13.1 主要风险

| 风险 | 说明 | 应对 |
|---|---|---|
| 需求不清 | 用户自然语言描述可能缺失关键规则 | Builder Agent 必须强制澄清 |
| 生成质量不稳定 | Agent 可能生成不合理模型或流程 | Business IR 校验 + 测试 + 人工确认 |
| 运行事故 | 自动生成系统可能影响真实业务 | 沙箱预览 + 灰度发布 + 回滚 |
| 权限错误 | 企业系统中权限错误风险高 | 权限矩阵可视化 + 权限测试 |
| 范围失控 | 企业业务系统范围极大 | MVP 聚焦资产盘点 / 仓储出入库 |
| 被认为只是低代码 | 市场认知可能偏差 | 强调 Agent 构建、Business IR、确定性运行和自维护 |
| 服务商使用门槛 | 实施服务商需要学习新方法 | 提供模板、访谈脚本、交付文档生成 |

### 13.2 产品原则

1. **AI 生成，但人确认后发布。**
2. **运行期优先确定性，不滥用 LLM。**
3. **所有变更必须有版本记录。**
4. **所有关键操作必须可审计。**
5. **先做好一个场景，再扩展多个行业。**
6. **优先服务懂业务流程的人，而不是追求所有员工自助搭建。**

## 14. 当前最小产品定义

第一版 ANC 可以定义为：

> 一个面向企业数字化负责人和数字化实施服务商的 AI 应用生成与业务流程运行平台。用户通过 Builder Agent 描述资产盘点 / 仓储出入库需求，系统生成可运行的 Web 管理后台、H5 扫码操作端、数据模型、流程、权限、日志和审计，并支持后续通过自然语言修改和版本发布。

第一版成功标准：

1. 能从自然语言生成结构化业务规格；
2. 能生成并运行一个完整的资产 / 仓储应用；
3. 能支持入库、出库、盘点三条核心流程；
4. 能生成基础权限和审计日志；
5. 能根据用户反馈修改应用并生成新版本；
6. 能让数字化实施人员看到交付效率提升的可能性。
