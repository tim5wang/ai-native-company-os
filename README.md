# ANC（Ai Native Company，AI原生公司）

ANC 的概念是仿 OPC 生搬硬造出来的名词，意味着具有比 OPC基建更强的基建系统，在AI原生的平台上构建起一个公司的内部系统。

产品定位： 企业级的AI Agent Native的编排系统，是一人公司或者多人公司的AI基建系统，由ANC连接和编排一切流程和应用。

落地场景：帮传统企业做数字化转型，首先以比较低的成本完成传统企业的流程数字化，节约信息流转成本，之后让企业员工的多数固定流程的工作可以由AI代练。

企业用户基于ANC搭建起应用形式包括：APP网页客户端应用(比如客人点菜端、员工清点资产客户端、仓储出入库客户端)，APP客户端应用(应用同APP网页客户端)，WEB客户端应用(仓储管理后台、考勤后台等等)，PC cli应用(开发者Agent，运维Agent等等)，云服务后台应用(数据大盘等)，知识库问答应用(如文档客服、产品客服)。

应用创建和更新应当是敏捷的：管理员权限的用户给Builder Agent提出需求，Builder Agent做需求澄清后完成应用的构建或者更新。

应当有良好的可视化：可视化的看到流程是怎么流转的，数据是怎么流转的，应用在线运维面板等等。

AI辅助运维：AI可以帮忙完成问题定位、基于在线日志采样的问题发现、基于在线监控的问题发现和自主响应。

自下而上：
    底层： 系统首先拥有一套完善的可以被AI治理和理解的底层基建，包括但是不限于: 数据库、缓存、消息队列、权限管理、应用管理和审计、用户管理、对象管理、文件系统、日志平台、邮件系统、监控平台、重型应用运行容器、轻量级应用的沙箱、基础的UI组件、IM组件、APP容器组件、小程序组件、App化的Admin管理入口、应用构建Agent等等，这些基建不一定要从零开始开发，可以基于已有的成熟开源软件做二次开发或者wrap.

    应用层： 之后在此之上按照企业定制化实现一些资产管理、OA流程、面向客户的轻应用、带有AI能力的应用、企业定制化的管理后台等等，这些定制不是让AI毫无章法的从零写代码，而是基于基建开发，从而保证系统的可治理、可观测、可AI维护。

    AI智能层：现在时髦的AI应用，比如floatboat ai已经可以做到"代练"了，当OPC者用 floatboat 一段时间后，floatboat可以理解用户在不同应用之间的工作流程，从而构建起完整的用户行为上下文，从而可以在用户离线时，接管操作，实现替老板打工。本平台短期不需要，但是长期需要这样的能力，这对ANC平台搭建出来的 app的可观测性有比较高的要求，不然后期平台知道应用流程，但是不知道用户是如何使用应用的。

自上而下：
    用户体验很重要，可观测、可解释，非开发者用户也能搭建的流程和原子能力，让懂得业务流程的人不必懂AI就能搭建起企业级可用的应用来。
    即使平台的AI做的很完善，基建很好理解，基于基建搭建起的应用有文档沉淀，有流程记录，但是to B 的企业人员，构建起的协作流程未必好理解，这就需要平台的基础组件有丰富的数据上报，从而方便观测用户行为，方便AI辅助用户完成工作任务。
