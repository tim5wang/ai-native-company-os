# ANC（AI Native Company，AI 原生公司）

ANC 是面向企业数字化与 AI 落地的 AI 原生业务操作系统。它的目标不是简单生成一次性应用，而是提供一套可治理、可观测、可复用的企业级 AI 基建，让企业内部流程、数据、应用和 Agent 能被统一编排与持续演进。

## 产品定位

ANC 是企业级 AI Agent Native 编排系统，可作为一人公司或多人公司的 AI 基建。它连接和编排企业内部的流程、应用、数据、权限、审计、运维和 AI Agent。

典型落地路径：先以较低成本帮助传统企业完成流程数字化，降低信息流转成本；再逐步让企业员工的大量固定流程工作可以由 AI 辅助执行或代练。

## 典型应用形态

企业可以基于 ANC 构建多种应用：

- H5 / Web 操作端：点餐、资产清点、仓储出入库、移动审批等；
- Web 管理后台：仓储管理、资产管理、考勤管理、流程管理等；
- App / 小程序容器应用：面向员工或客户的轻应用；
- PC / CLI 应用：开发者 Agent、运维 Agent 等；
- 云服务后台：数据大盘、监控、报表、调度服务等；
- 知识库问答应用：文档客服、产品客服、内部助手等。

## 核心理念

### 1. AI 构建，确定性运行

管理员或实施顾问向 Builder Agent 描述需求，Agent 进行需求澄清，并生成应用、数据模型、页面、流程、权限和测试。应用运行时应尽量依赖确定性代码、规则和工作流，而不是每一步都调用 LLM。

### 2. 模块复用，而不是每次从零生成

ANC 不应只沉淀行业模板，还需要系统化的业务模块管理能力。

为第一家公司完成数字化和 AI 落地后，其中的大部分能力应该沉淀为可复用模块；为第二家公司交付时，应优先复用、升级或修订这些模块，而不是重新生成。这样可以缩短构建周期，降低构建阶段的 LLM 调用成本，并提升交付质量稳定性。

业务模块应支持：

- 独立版本管理；
- 模块依赖声明；
- 模块组合；
- 参数化配置；
- 定制扩展；
- 升级与回滚；
- 私有或公开的模块注册中心。

例如，一个仓储 / 资产管理应用可以拆成物料模型、仓库模型、入库流程、出库流程、盘点流程、库存流水、扫码操作、权限模板、通知规则等模块。不同客户之间优先复用这些模块，只在差异点上做定制。

### 3. 可治理、可观测、可 AI 维护

基于 ANC 构建的应用不应是 AI 随意生成的黑盒代码，而应建立在统一基建和中间表示之上，确保系统可治理、可观测、可审计、可回滚，并能被 AI 理解和维护。

平台需要提供良好的可视化能力，让用户能看到流程如何流转、数据如何流转、应用如何运行，以及 Agent 做了哪些生成、修改和诊断操作。

## 分层架构

### 底层基建

ANC 底层需要一套可被 AI 理解和治理的企业级基础设施，包括但不限于：数据库、缓存、消息队列、权限管理、应用管理、审计、用户管理、对象管理、文件系统、日志平台、邮件系统、监控平台、应用运行容器、轻量级沙箱、基础 UI 组件、IM 组件、App / 小程序容器、Admin 管理入口和应用构建 Agent。

这些基建不一定全部从零开发，可以基于成熟开源软件进行二次开发或封装。

### 应用层

在底层基建之上，ANC 支持按企业需求构建资产管理、OA 流程、客户轻应用、AI 应用、企业管理后台等。应用构建应优先基于已有模块、组件和业务中间表示进行组合与定制，而不是让 AI 从零无约束写代码。

### AI 智能层

短期内，AI 主要负责需求澄清、应用构建、修改、诊断和运维辅助。长期来看，ANC 应支持更强的代练能力：通过应用可观测性和用户行为上下文，理解员工在不同应用间的工作流程，并在授权范围内辅助或接管固定流程任务。

## 用户体验原则

- 非开发者也能描述和搭建业务流程；
- 懂业务的人不必懂 AI，也能构建企业级可用应用；
- 流程、数据、权限、版本和 Agent 操作都应可解释；
- AI 生成结果需要可预览、可测试、可审计；
- 关键变更必须人工确认后发布；
- 系统应持续沉淀业务模块，形成越交付越快、越交付越便宜的复用飞轮。

## AI 辅助运维

ANC 应支持 AI 辅助问题定位、日志分析、监控异常发现、流程瓶颈识别和修复建议生成。AI 不应默认直接修改生产系统，而应先生成诊断结果、影响分析和变更草案，再由有权限的用户确认发布。
